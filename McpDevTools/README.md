# Development_McpDevTools

[![Packagist](https://img.shields.io/packagist/v/jeanmarcos/module-mcp-dev-tools.svg)](https://packagist.org/packages/jeanmarcos/module-mcp-dev-tools)

> вљ пёЏ **FOR LOCAL DEVELOPMENT ONLY вЂ” NEVER ENABLE IN PRODUCTION**

Exposes a [Model Context Protocol](https://modelcontextprotocol.io) (MCP) server so LLM clients like **Claude Code**, **Cursor**, **Cline** or **Continue** can introspect a running Magento installation: list modules, find plugins on a class, resolve a class to its file path / interceptor / preferences / virtual types, look up routes and observers, read config, identify the owning module of any class.

Read-only in v0.1. Designed for **dev introspection** вЂ” what plugins fire on `Magento\Catalog\Model\Product::getPrice`, which observers listen to `sales_order_place_after`, what module owns `vendor/magento/.../File.php` вЂ” *not* for store operations (no product/order/customer mutations).

---

## What it does

Two transports, one tool registry:

- **stdio** (primary): `bin/magento mcp:serve` blocks on STDIN, hands the streams to the MCP SDK, responds on STDOUT. Magento bootstraps once; subsequent tool calls reuse the warm object manager (sub-millisecond after warm-up).
- **HTTP** (experimental, single-shot only in v0.1): `POST /dev-mcp` with a JSON-RPC `tools/call` envelope. Loopback-bound by default; `X-MCP-Token` header required. **No `initialize` ceremony, no session continuity** вЂ” for those, use stdio. v0.2 graduates this to full Streamable HTTP with sessions.

### Tools (v0.1)

| Tool | Args | Returns |
|---|---|---|
| `list_modules` | вЂ” | Every installed module: name, setup_version, sequence, active state. |
| `find_plugins` | `class`, optional `method` | Plugins attached to a class (chain order via public API when `method` set; full index via reflection when not). |
| `find_observers` | `event` | Observers registered for a Magento event (instance, method, shared/disabled). |
| `resolve_class` | `name` | File path, generated interceptor, preferences resolving here, virtual types pointing here, owning module. |
| `list_routes` | optional `area` (frontend/adminhtml/webapi_rest/webapi_soap/graphql) | Every route: route_id, frontName, modules. |
| `get_config` | `path`, optional `scope`, `scope_code` | Resolved config value with proper scope. |
| `which_module` | `query` (FQCN or filesystem path) | Owning module/theme/library, registered path, expected file path. |
| `describe_di` | `class` | Constructor args with types, preferences here, virtual types here, plugins indexed. |

`describe_di` and `which_module` are the two highest-density single calls вЂ” most LLM questions about a Magento class can be answered by one of those.

## Safety model

| Mode | `Allow in Production` flag | Behavior |
|---|---|---|
| `developer` / `default` | any | **active** вЂ” server runs |
| `production` | `No` (default) | **inactive** вЂ” `mcp:serve` exits 1; `/dev-mcp` returns 503 |
| `production` | `Yes` | **active** вЂ” explicit override (use at your own risk) |

Implementation: shared `Development\Core\Model\ProductionGuard` from [`jeanmarcos/module-core-local-development`](https://packagist.org/packages/jeanmarcos/module-core-local-development), wired via a `virtualType` in `etc/di.xml` to the config path `development/mcp_dev_tools/allow_in_production`.

The HTTP transport additionally requires:

- `X-MCP-Token` header matching `development/mcp_dev_tools/auth_token` (auto-generated on install via setup patch; rotate by clearing the field and running `setup:upgrade`).
- Loopback origin (`127.0.0.1` / `::1`) when `Loopback Only` is on (default).

The stdio transport is not gated by token вЂ” it's a subprocess of the LLM client and inherits the user's privileges.

## Install

```bash
composer require --dev jeanmarcos/module-mcp-dev-tools
bin/magento module:enable Development_McpDevTools
bin/magento setup:upgrade
bin/magento setup:di:compile
bin/magento cache:flush
```

The auth token is generated automatically by a setup data patch on first `setup:upgrade`.

### Wire into Claude Code (stdio)

```bash
claude mcp add magento-dev "$(pwd)/bin/magento mcp:serve"
```

Then in any Claude Code session inside that project:

> *"Using the magento-dev MCP, what plugins are attached to `Magento\Catalog\Model\Product::getPrice`?"*

Claude calls `find_plugins` and gets a structured answer instead of grep-walking `vendor/`.

### Wire into Cursor (stdio)

Edit `~/.cursor/mcp.json`:

```json
{
  "mcpServers": {
    "magento-dev": {
      "command": "/absolute/path/to/magento/bin/magento",
      "args": ["mcp:serve"]
    }
  }
}
```

### Use the HTTP transport (experimental)

```bash
TOKEN=$(bin/magento config:show development/mcp_dev_tools/auth_token)
curl -sS http://127.0.0.1/dev-mcp \
    -H "X-MCP-Token: $TOKEN" \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"list_modules","arguments":{}}}'
```

## Compatibility & known fragility

- Magento 2.4.x (tested against 2.4.4, 2.4.6, 2.4.7).
- PHP 8.1+, ext-json.
- Depends on `logiscape/mcp-sdk-php: ^1.7` and `jeanmarcos/module-core-local-development: ^1.0` (installed automatically by Composer).
- **`find_plugins` without `method` arg uses reflection** on `\Magento\Framework\Interception\PluginList\PluginList::$_data`, an internal property. The shape can change in any Magento minor release. When it does, fall back to passing the `method` arg (uses the public, stable `getNext()` API).

## Roadmap

- **v0.2**: HTTP transport graduates to full **Streamable HTTP** with `Mcp-Session-Id` sessions and the `initialize` handshake. CIDR allowlist for non-loopback. New tools: `list_config_paths(pattern)`, `list_preferences(interface_filter)`.
- **v0.3**: `trace_request(url)` вЂ” runs a synthetic request and returns the plugins/observers/queries that fired during it.
- **v0.4**: Mutating tools (`flush_cache`, `reindex`, `set_config`) gated behind a separate `allow_mutations` flag, never enabled in production even with the bypass guard on.
